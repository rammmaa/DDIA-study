## [1] 신뢰성

- 결함(fault) 과 장애(failure) 를 구분.
    - 결함(fault) = 시스템 안의 한 부분이 고장 나거나 이상해진 상태
    - 장애(failure) = 그 때문에 사용자에게 필요한 서비스 전체를 못 주는 상태
    - 내결함성 / 탄력성 = 결함이 생겨도 시스템 전체는 계속 돌아가게 하는 능력
- 좋은 시스템은 결함이 생겨도 그것이 곧바로 장애로 번지지 않게 만드는 것.
- 내결함성 시스템이라면, 일부러 문제를 만들어보는 게 의미 있음.
왜냐하면 실제로 심각한 문제의 상당수는 결함 자체보다, 그 결함을 처리하는 코드가 허술해서 더 커지기 때문임.
실제로 계속 훈련하고 테스트해야 함.
    - 넷플릭스의 카오스몽키
- 보안 문제는 예방이 더 중요함. 
어떤 결함은 나중에 복구하면 되지만,
보안 사고는 한 번 당하면 되돌릴 수 없는 경우가 많기 때문임.

### (1) 하드웨어 결함

- 하드웨어는 고장은 흔함. 
좋은 시스템은 고장이 나도 계속 돌아가게 중복과 내결함성으로 설계함.
- 개별 장비 신뢰성 ≠ 전체 시스템 신뢰성.
부품 하나는 오래 버텨도, 개수가 많으면 매일 뭔가 죽음.
- 기본 대응은 중복(redundancy) 임.
- 안 죽는 서버 보다 죽어도 되는 시스템이 중요함.

### (2) 소프트웨어 오류

- 소프트웨어 오류는 하드웨어 고장과 성격이 다름.
    - 소프트웨어 오류는 체계적(systematic) 이라서, 같은 버그가 여러 서버에서 동시에 같은 방식으로 터질 수 있음.
- 이런 버그는 평소엔 숨어 있다가 특수한 상황에서 터짐.
    - 소프트웨어는 “환경이 이럴 것이다”라는 가정 위에 서 있는데, 그 가정이 언젠가 깨질 수 있음.
    - 이런 종류의 버그는 보통 오랫동안 안 보이다가,
    이상한 입력, 드문 상황, 예상 못 한 상호작용이 생겼을 때 갑자기 드러남.
- 소프트웨어 오류는 버그 1개 고치면 끝이 아니고,
설계, 테스트, 격리, 모니터링으로 꾸준히 대비하고 피해를 줄여야 하는 문제임.

### (3) 인적 오류

- 사람 실수는 큰 장애 원인임.
- 시스템을 사람이 실수한다는 전제로 만듦.
- 인터페이스를 잘 설계해 실수를 줄여야 함.
    - 잘 설계된 추상화, API, 관리 인터페이스로 옳은 일은 쉽게, 잘못된 일은 어렵게 만듦.
- 실수하기 쉬운 곳과 실제 서비스 영향을 분리해야 함.
    - 사람이 가장 많이 건드리고 실수할 수 있는 부분은, 실제 사용자에게 바로 피해가 가지 않게 분리하는 게 중요함.
    - 실제 데이터로도 실험할 수 있지만 실제 사용자에게 영향은 없는 비프로덕션 샌드박스 (테스트용 공간) 를 제공해야 함.
- 테스트/ 롤백/ 점진적 배포로, 복구 가능성을 높여야 함.

### (4) 신뢰성의 중요도

- 신뢰성은 대부분의 소프트웨어가 기본적으로 고민해야 할 문제임.
평범한 앱도 사용자 시간, 돈, 데이터, 기억, 신뢰를 맡고 있기 때문임.
단, 현실에서는 비용 때문에 신뢰성을 일부 희생할 수도 있으니, 그때는 무엇을 포기하는지 자각한 상태에서 결정해야 함.

## [2] 확장성

- 지금 안정적이어도, 사용자와 데이터가 늘면 성능이 나빠질 수 있음.
- 성능 저하의 흔한 원인은 부하 증가임.
    - 동시 사용자 수, 요청량, 데이터 양
- 확장성: 늘어난 부하를 감당하는 능력
    - 커진 규모에서도 계속 동작하게 만드는 능력
- 확장성은 단순히 “있다/없다”로 말할 수 없음.
    - 무엇이 얼마나 늘어나는지에 따라 대응 방식이 달라지므로, 
    상황 없이 “확장 가능”이라고 말하는 건 의미가 약함.
- 성장 일어날 때 어떻게 버틸 것인가
    - 어떤 자원을 더 넣을지
    - 어떤 구조로 바꿀지

### (1) 부하 기술하기

- 확장성이란 단순히 서버를 더 붙이는 문제가 아니라, 
어떤 부하가 진짜 병목인지 찾고, 읽기와 쓰기 비용을 어디에 배분할지 설계하는 문제.
- 확장성을 보려면 부하를 숫자로 표현해야 함.
    - 초당 요청 수, 읽기/쓰기 비율, 활성 사용자 수
- 중요한 부하 매개변수는 시스템마다 다름.
    - 트위터에서는 단순 트윗 수보다 팔로워 수 분포가 더 중요했음.
- 읽기 때 계산할지, 쓰기 때 미리 계산할지 선택해야 함.
    - 접근 방식 1: 쓰기 싸고 읽기 비쌈
    - 접근 방식 2: 쓰기 비싸고 읽기 쌈
- 트위터는 읽기가 훨씬 많아서 쓰기 시점에 더 일하는 방식이 유리했음.
    - 그래서 홈 타임라인 캐시 방식으로 옮겼음.
- 평균값만 보면 안 됨.
    - 유명인처럼 팔로워가 극단적으로 많은 사용자가 전체 설계를 바꿀 정도의 병목을 만들 수 있음.
- 현실에서는 하이브리드가 자주 답.
    - 모든 사용자에게 한 가지 방식만 적용하는 게 아니라, 
    일반 사용자와 유명인을 다르게 처리함.

### (2) 성능 기술하기

- 성능 기술하기 2가지
    - 부하만 늘리고 자원은 그대로면, 성능이 얼마나 나빠지는가
    = 얼마나 느려지나
    - 성능을 그대로 유지하려면, 자원을 얼마나 더 넣어야 하는가
    = 안 느려지려면 서버 얼마나 더 넣어야 하나
- 처리량 vs 응답 시간 (시스템 종류에 따라 중요한 성능 지표가 다름)
    - 일괄 처리(batch) [얼마나 많이/ 빨리 처리하냐]
        - 하둡 같은 시스템에서는 처리량(throughput) 이 중요함.
        - 초당 몇 건 처리하냐
        - 이 데이터 작업을 끝내는 데 총 몇 분/몇 시간이 걸리냐
    - 온라인 시스템 [사용자가 얼마나 오래 기다리냐]
        - 웹서비스, 앱 API 같은 데서는 응답 시간(response time) 이 더 중요함.
        - 사용자가 요청을 보낸 순간부터 답을 받을 때까지 얼마나 기다렸는가
- 그래서 평균보다 백분위를 봐야 한다
    - 평균(mean) 응답 시간은 전형적인 사용자 경험을 잘 보여주지 못함.
    - 평균은 느린 몇 개에 크게 흔들릴 수 있고, 실제로 얼마나 많은 사용자가 느린 응답을 겪었는지를 잘 말해주지 못하기 때문임.
    - 그래서 더 좋은 지표가:
    - 중앙값(median, p50)
    95백분위(p95)
    99백분위(p99)
    99.9백분위(p999)
    - 백분위 = “느린 요청들이 얼마나 느린가”까지 보여줌
- 왜 p95, p99 같은 “느린 꼬리”가 중요하냐
    - tail latency, 즉 상위 백분위의 느린 응답 시간이 사용자 경험에 직접 영향을 줌.
    - 사용자는 평균을 체감하지 않고, 자기가 겪은 느린 순간을 기억하기 때문임.
    - 그래서 큰 서비스들은 보통 median만 보지 않고 p95, p99, p999를 중요하게 봄.
- SLA/SLO에 왜 백분위를 쓰나
    - 백분위가 서비스 수준 목표(SLO) 나 서비스 수준 협약(SLA) 에 자주 쓰임.
    - 서비스가 “빠르다”를 감으로 말하는 게 아니라 숫자로 계약하고 약속하는 것.
    - 예:
    - 중앙값 응답 시간 < 200ms
    - p99 < 1초
    - 가용성 99.9% 이상
- 큐 대기와 선두 차단(head-of-line blocking)
    - 서버는 동시에 처리할 수 있는 양이 제한돼 있음.
    - 선두 차단(head-of-line blocking):  몇 개의 느린 요청이 앞에 있으면, 뒤의 빠른 요청도 같이 기다려야 함.
    - 느린 요청 몇 개가 시스템 전체 체감을 망칠 수 있음. 전체 응답 시간이 길어져서.
- 왜 클라이언트 쪽에서 응답 시간을 재야 하나
    - 사용자 체감 성능을 알려면 클라이언트 기준 측정이 중요함.
    - 서버 안에서만 측정하면 
    실제 사용자가 느끼는 네트워크 시간, 큐 대기 시간, 앞 요청 때문에 밀린 시간 같은 게 덜 보일 수 있음.
- 부하 테스트할 때 왜 “응답 기다리지 말고 계속 보내라” 하나
    - 인위적으로 부하를 만들 때, 부하 생성 클라이언트는 응답 시간과 독립적으로 계속 요청을 보내야 함.
    - 왜냐하면 이전 요청 응답을 다 기다렸다가 다음 요청을 보내면, 실제보다 큐가 덜 쌓이고 시스템이 덜 바쁜 것처럼 보이기 때문임.
    - 그런 테스트는 너무 낙관적이라서 실제보다 좋아 보이는 결과를 냄.
- 백분위는 어떻게 계산하고 모니터링하나
    - 모니터링 대시보드에 백분위를 넣으려면 최근 10분 같은 롤링 윈도에서 계속 계산해야 함.
    - 근데 모든 응답 시간을 다 들고 있다가 계속 정렬하는 건 비효율적일 수 있으니,
    - forward decay
    t-digest
    HdrHistogram
    같은 근사 알고리즘을 쓴다고 합니다.
    - 포인트: 백분위의 평균을 내면 안 됨.
    예: 여러 서버의 p99를 그냥 평균내는 건 수학적으로 의미가 약함.
    올바른 집계는 보통 히스토그램을 합치는 방식.

### (3) 부하 대응 접근 방식

- 현재 부하에 맞는 구조가 미래 부하에도 맞는 건 아님.
    
    부하가 10배 커지면 아키텍처도 다시 봐야 할 수도.
    
- 확장에는 두 큰 방향
    
    수직 확장(더 좋은 한 대)과 수평 확장(여러 대로 분산)
    
- 좋은 구조는 보통 한 방식만 쓰지 않음.
    
    현실에서는 수직/수평 확장을 실용적으로 섞는 경우
    
- 자동 확장과 수동 확장은 장단점이 다름
    
    자동 확장은 예측 어려운 부하에 유리하고, 
    수동 확장은 더 단순하고 예측 가능할 수 있습니다.
    
- 상태 없는 서비스는 확장하기 쉽고, 상태 있는 데이터 시스템은 어려움.
    
    웹 서버 여러 대 띄우는 건 비교적 쉽지만, 
    DB를 여러 대로 나누는 건 훨씬 복잡함
    
- 모든 서비스에 통하는 만능 확장 구조는 없음
    
    읽기량, 쓰기량, 데이터 크기, 응답 시간 요구, 접근 패턴에 따라 설계가 달라짐
    
- 확장 설계는 “어떤 부하가 올 것인가”라는 가정 위에 세우는 것

## [3] 유지보수성

- 소프트웨어 비용의 큰 부분은 초기 개발보다 유지보수에 들어감
- 좋은 시스템은 처음 만들기 쉬운 시스템이 아니라, 나중에 운영·수정·확장하기 쉬운 시스템임

### (1) 운용성

- 운영팀은 단순 감시만 하는 게 아니라,
모니터링, 복구, 원인 분석, 패치, 용량 계획, 문서화까지 맡음.
- 좋은 운용성을 가진 시스템은
가시성, 자동화 지원, 장비 독립성, 문서, 예측 가능성, 적절한 수동 제어 수단을 제공함.

### (2) 단순성

- 복잡한 시스템은 유지보수를 망치므로, 
기능을 줄이지 않더라도 구현에서 생긴 불필요한 복잡함을 없애고, 
좋은 추상화로 시스템을 이해하기 쉬운 부품들로 나눠야 함.
- 복잡함은 유지보수의 적
    
    복잡한 시스템은 이해하기 어렵고, 고치기 어렵고, 버그도 잘 생김
    
- 단순성은 핵심 목표
    
    좋은 시스템은 단순성을 중요하게 생각해야 함. 
    단순성은 사용자 화면이 아니라 시스템 내부 구조의 단순함.
    
- 단순하게 만든다고 기능을 줄인다는 뜻은 아님
    
    핵심은 불필요한 복잡함, 즉 우발적 복잡도를 줄이는 것.
    
- 우발적 복잡도: 구현 때문에 생긴 복잡함
    
    문제 자체가 원래 어려운 게 아니라, 구현을 못 해서 복잡해진 부분은 줄여야 함.
    
- 추상화가 가장 강력한 해결 도구
    
    좋은 추상화는 복잡한 내부를 숨기고, 이해하기 쉽고 재사용 가능한 구조를 만듦.
    
- 좋은 추상화를 찾는 건 어렵지만 매우 중요
    
    특히 큰 시스템과 분산 시스템에서는 이게 유지보수성을 좌우.
    

### (3) 발전성

- 시스템 요구사항은 계속 바뀌므로, 좋은 시스템은 처음부터 변화에 적응하기 쉽게 설계되어야 하며, 이런 능력을 발전성(evolvability)이라 함.
- 요구사항은 거의 반드시 바뀐다
    
    시스템은 시간이 지나면서 새 기능, 새 사용 사례, 새 규제, 새 플랫폼, 규모 성장 같은 변화
    
- 그래서 시스템은 “변화 가능성”을 전제로 설계해야 함
    
    처음 상태에만 맞는 구조가 아니라, 나중에 수정과 확장이 가능한 구조가 중요
    
- 애자일, TDD, 리팩터링은 변화 대응에 도움이 됨
    
    하지만 이들은 보통 코드 파일 몇 개 수준의 작은 범위 변화에 더 초점을 둠
    
- 더 큰 수준의 변화
    
    여러 서비스와 데이터 시스템 전체 수준에서 아키텍처를 어떻게 바꾸기 쉽게 만들까
    
- 발전성은 단순성과 추상화와 연결됨
    
    간단하고 이해하기 쉬운 시스템일수록 나중에 고치고 바꾸기 쉬움
