# [1] 관계형 모델과 문서 모델

- “관계형 모델이 왜 그렇게 오래 기본값처럼 쓰였는가?”
- 관계형 DB는 범용성이 강했다

## (1) NoSQL 의 탄생

- Not Only SQL
- NoSQL이 SQL을 이겼다”가 아님
- **다중 저장소 지속성(polyglot persistence)**
- **애플리케이션마다 요구사항이 다르므로, 관계형 DB와 다양한 비관계형 저장소가 함께 쓰일 것**

- **대규모 데이터와 매우 높은 쓰기량**을 더 잘 처리하고 싶어서
- 오픈소스
- **관계형 모델이 잘 지원하지 않는 특수 질의**가 필요
- 고정된 스키마가 답답해 더 유연한 데이터 모델 필요

## (2) 객체관계형불일치

- 애플리케이션 코드의  객체지향 언어 vs 관계형 DB
- ORM은 완전한 해결은 아님
    - ActiveRecord, Hibernate 같은 ORM은: SQL 매핑 코드 (번역 작업)을 자동으로 좀 도와주는 도구
    - 왜냐하면 근본적으로 객체는 중첩 구조, 리스트, 하위 객체가 자연스럽고
    - 관계형 모델은 표와 조인이 자연스럽기 때문
- 링크드인 프로필 (이력서) 예시
    - 일대다 구조
- 관계형 DB에서는 이걸 어떻게 표현 방식 3가지
    - 한 사람 정보를 여러 테이블에 나눔
    - 한 행 안에 구조화 데이터 넣기
        - SQL DB는 XML, JSON, 구조화 데이터 타입을 어느 정도 지원해서
        - 한 행 안에 다중 값을 넣고 내부를 질의하거나 인덱싱하는 기능도 일부 제공
        - Oracle, DB2, SQL Server, PostgreSQL, MySQL 같은 DB들이 이런 기능을 서로 다르게 지원
    - 그냥 텍스트 칼럼에 JSON/XML 통째로 넣기
        - 예를 들어 `profile_json` 같은 텍스트 칼럼에 JSON 문자열을 통째로 저장하고,
        애플리케이션이 꺼내서 해석하는 방식
        - 보통 **DB가 그 안쪽 내용을 일반적으로 잘 질의하지 못한다**는 문제
- 결론은 문서 모델(JSON)이 더 자연스럽긴 함
    - **한 사람 프로필 한 묶음**
    - JSON은 그 묶음을 그대로 표현하기 쉬움
    - 앱 코드가 다루는 객체 모양과도 꽤 비슷
    - 그래서 어떤 개발자들은 JSON 모델이 앱 코드와 저장 계층 사이의 불일치를 줄여준다고
- locality (**관련된 정보가 한곳에 모여 있다**는 뜻)
    - JSON 표현이 multi-table 스키마보다 더 나은 locality를 가진다
    - 관계형 예시에서는 프로필 하나를 가져오려면:
    - users 조회+ positions 조회+ education 조회+ contact_info 조회 처럼 여러 번 조회하거나, 
    아니면 여러 테이블을 조인
- 트리구조
    - 즉, **일대다 관계가 트리처럼 생겼다**는 뜻
        - 예를 들어:
            - 사용자
                - 경력들
                    - 경력1
                    - 경력2
                - 학력들
                    - 학력1
                    - 학력2
                - 연락처

## (3) 다대일과 다대다 관계

- 문서 모델은 한 사람 중심의 트리형 데이터에는 잘 맞지만, 
지역·업계·회사·학교·추천인처럼 여러 문서가 서로 연결되기 시작하면 
관계형 모델의 조인과 정규화가 훨씬 중요해진다.

- ID를 쓰는 이유는 **중복 제거와 정규화** 때문이다.
    - 텍스트를 반복 저장하지 않고, 의미 있는 정보는 한 곳에만 둔다.
- 이렇게 하면 **다대일 관계**가 생긴다.
    - 많은 사용자가 하나의 지역/업계를 참조한다.
- 관계형 DB는 조인이 쉬워서 이런 참조 구조를 잘 처리한다.
    - 문서 DB는 조인 지원이 약해 앱 코드가 더 많은 일을 해야 할 수 있다.
- 기능이 늘면 회사, 학교, 추천인처럼 데이터가 점점 더 연결된다.
    - 즉, 구조가 단순 트리에서 벗어난다.
- 그래서 **문서 모델은 초기엔 자연스러워도, 관계가 많아질수록 불편해질 수 있다**.
    - 바로 이 지점에서 관계형 모델의 장점이 다시 드러난다.

## (4) 문서 데이터베이스는 역사를 반복하고 있나?

- 문서 DB는 JSON 같은 트리 구조를 자연스럽게 다루지만, 
그 구조가 과거의 계층형 DB와 비슷한 만큼, 다대다 관계와 조인 문제도 옛날과 비슷하게 다시 마주칠 수 있다.

- 관계형 DB는 다대다 관계와 조인을 잘 다룬다.
- 문서 DB/NoSQL은 이 문제를 다시 새롭게 고민하게 만들었다. <중첩트리구조>
- 그런데 이런 고민은 사실 새롭지 않다.
- 1960~70년대의 **IMS라는 계층형 DB** 도 JSON 문서 모델과 비슷한 구조를 가졌다.
- 그 DB도 **일대다에는 강했지만, 다대다와 조인에는 약했다.**
    - 그래서 개발자는 **중복 저장**하거나 **참조를 수동 처리**해야 했고,
    이 문제는 오늘날 문서 DB에서도 비슷하게 나타날 수 있다.
- 그래서 결국 관계형 모델이 강해졌다.

### 1)  네트워크모델

- 왜 결국 관계형 모델에 밀렸는지
- **네트워크 모델은 계층 모델보다 관계 표현은 풍부했지만, 
데이터를 찾는 경로를 프로그래머가 직접 관리해야 해서 코드가 너무 복잡하고 바꾸기 어려웠다**

- 네트워크 모델은 계층 모델을 확장해서
**한 레코드가 여러 부모를 가질 수 있게** 만든 모델
- 그래서 계층 모델보다
**다대일, 다대다 관계를 더 잘 표현**할 수 있
- 하지만 레코드 연결 방식이 **외래 키보다 포인터에 가까워서**, 
원하는 데이터에 가려면 **접근 경로**를 직접 따라가야 했음
- 이 방식은 하드웨어가 약하던 시절엔 효율적일 수 있었지만,
질의 코드와 갱신 코드가 너무 복잡하고 유연하지 못했음
- 특히 새로운 요구사항이 생겨 다른 방식으로 데이터를 찾고 싶어지면,
기존 코드까지 대거 수정해야 해서 **데이터 모델 변경이 어려웠음.**

### 2) 관계형모델

- 왜 관계형 모델이 결국 강해졌는가
- 네트워크 모델/계층 모델은 데이터를 따라가는 **경로**가 중요
반면 관계형 모델은 **“경로는 신경 쓰지 말고, 그냥 어떤 데이터가 필요한지만 말해라”**

- **관계형 모델의 강점은 데이터를 복잡한 연결 경로로 따라가지 않아도 되고, 
원하는 조건만 말하면 DB가 알아서 가장 적절한 접근 방법을 고른다**는 점
    - 데이터를 따라가는 길을 개발자가 직접 코딩하지 않아도 되고, DB가 질의 최적화기를 통해 가장 좋은 접근 경로를 자동으로 골라준

- **관계형 모델은 데이터를 표처럼 펼쳐서 다룬다.**
복잡한 중첩 구조나 수동 접근 경로에 의존하지 않는다.
- **원하는 데이터는 조건으로 찾는다.**
어떤 경로로 갈지를 개발자가 직접 정하지 않아도 된다.
- **질의 최적화기(query optimizer)가 접근 경로를 자동 결정한다.**
조인 순서, 사용할 인덱스 같은 실행 계획을 DB가 알아서 정한다.
- **새 인덱스를 추가해도 질의를 크게 고칠 필요가 없다.**
그래서 기능 추가와 성능 개선이 더 유연하다.
- **최적화기는 만들기 어렵지만, 한 번 잘 만들면 모든 애플리케이션이 혜택을 본다.**
이게 관계형 모델의 장기적 강점이다.

### 3) 문서 데이터베이스와의 비교

- 문서 DB는 중첩 문서 구조 때문에 계층 모델과 닮았지만, 
관계가 복잡해지면 결국 관계형 DB처럼 다른 항목을 ID로 참조해야 하므로 
둘의 차이가 생각만큼 절대적이지는 않음
- 문서 DB는 한 문서 안에 관련 정보를 중첩해서 담기 때문에 일대다 구조에 자연스럽다.
- 관계가 복잡해져서 다대일 다대다가 생기면, 문서 DB도 결국 **다른 대상을 ID로 참조**해야 한다.
- 문서 DB와 관계형 DB는 이 구간에서는 생각보다 완전히 다른 모델이 아니다.
이름이 다를 뿐, 둘 다 “참조를 읽을 때 해석”하는 구조를 쓴다.

## (5) 관계형 데이터베이스와 오늘날의 문서 데이터베이스

- 문서형은 한 덩어리 데이터를 유연하게 저장하고 읽는 데 강하고, 
관계형은 서로 연결된 데이터를 다루는 데 강하다
- **데이터 모델 차이**
- 문서 DB의 장점은 **스키마 유연성**이다.
    - 문서 DB는 **관련 정보가 한곳에 모여 있어 읽기 성능이 좋을 수 있다.**
- 관계형 DB는 **조인과 복잡한 관계 처리**에서 더 강하다.

### 1) 어떤 데이터 모델이 애플리케이션 코드를 더 간단하게 할까?

- **무조건 문서 모델이 더 단순한 것도 아니고, 무조건 관계형 모델이 더 단순한 것도 아니다.
데이터끼리 어떤 관계를 맺고 있느냐에 따라 달라진다**
- **문서처럼 생긴 데이터**면 문서 모델이 더 자연스럽고
- **관계가 많이 얽힌 데이터**면 문서 모델은 불편해지고 관계형 모델이 더 무난해지며
- **아주 많이 연결된 데이터**는 그래프 모델이 더 자연스럽다

- **문서형 트리 구조라면 문서 모델이 더 자연스럽다.**
여러 테이블로 쪼개는 관계형 방식은 코드와 스키마를 불필요하게 복잡하게 만들 수 있다.
- **문서 모델에도 제한이 있다.**
문서 안의 깊은 중첩 항목을 직접 다루는 건 다소 불편할 수 있다.
- **조인이 약한 게 문제일 수도, 아닐 수도 있다.**
독립적인 이벤트 기록 같은 데이터는 조인이 거의 필요 없어서 문서형이 괜찮을 수 있다.
- **다대다 관계가 많아지면 문서 모델의 매력이 떨어진다.**
앱 코드가 중복 일관성 유지나 수동 조인을 떠맡게 될 수 있다.
- **어떤 모델이 더 단순한지는 관계 유형에 달려 있다.**
문서형 데이터면 문서 모델, 상호 연결이 많으면 관계형, 매우 연결이 많으면 그래프가 자연스럽다.

### 2) 문서 모델에서의 스키마 유연성

- **문서 모델의 스키마 유연성** 뜻
- 그게 **언제 장점이고 언제는 꼭 장점만은 아닌지**

- 문서 DB의 JSON은 보통 구조를 엄격히 강제하지 않아서 
바뀌는 데이터를 다루기 편할 수 있다.
- 하지만 “스키마가 없다”는 말은 완전히 정확하지 않고, 
읽는 쪽 코드에는 여전히 어떤 구조에 대한 가정이 있다

- 문서 DB의 JSON은 보통 구조를 강제하지 않아서 유연하다.
- 하지만 “스키마가 없다”기보다 “DB가 강제하지 않는다”가 더 정확하다.
- 그래서 문서 DB는 **읽기 스키마(schema-on-read)** 로 이해하는 게 좋다.
- 관계형 DB는 보통 **쓰기 스키마(schema-on-write)** 로 이해할 수 있다.
- 구조 변경이 잦거나 데이터가 제각각이면 문서형이 유리할 수 있지만, 구조가 비교적 일정하면 스키마 강제가 오히려 큰 장점이 된다.

### 3) 질의를 위한 데이터 지역성

- **“지역성(locality)”뜻**
- **그 장점에도 조건과 한계가 있다**

- 관련된 데이터가 물리적으로 서로 가까이 붙어 있으면 읽기 편하다
- **문서 DB의 지역성 장점은 “관련 데이터를 한 덩어리로 자주 읽을 때” 빛나지만, 
문서가 커지거나 부분 접근·부분 갱신이 많아지면 오히려 비효율적일 수 있다**

- 문서 DB는 관련 정보를 한 문서에 모아두기 때문에 **문서 전체를 자주 읽는 질의**에서 유리할 수 있습니다.
- 그 장점은 **문서의 큰 부분을 한 번에 읽을 때**에만 큽니다.
- 문서가 크면 작은 일부만 읽어도 전체를 불러와야 하고, 수정 때도 전체 재작성 비용이 생길 수 있습니다.
- 관련 데이터를 가까이 배치하는 지역성 아이디어 자체는 문서 DB만의 전용 무기가 아닙니다.

### 4) 문서 데이터베이스와 관계형 데이터베이스의 통합

- **관계형 DB와 문서 DB가 서로 경쟁만 하는 게 아니라, 
시간이 갈수록 서로 닮아가고 있다**
- **미래의 좋은 DB는 “관계형의 장점”과 “문서형의 장점”을 둘 다 가져가는 방향으로 가고 있다**

- **관계형 DB도 XML, JSON 같은 문서형 데이터를 점점 더 잘 지원한다.**
- **문서 DB도 조인이나 참조 해석 같은 관계형 기능을 일부 받아들이고 있다.**

# [2] 데이터를 위한 질의 언어

- **왜 SQL 같은 질의 언어가 중요한가**
- **왜 DB에서는 명령형보다 선언형이 유리한가**

- 명령형은 “어떻게 할지”를 하나씩 적는 방식이고,
- 선언형은 “무엇을 원한다”만 말하는 방식
- DB에서는 보통 선언형이 훨씬 유리
- **DB가 내부적으로 더 잘 최적화하고 병렬화할 수 있게 만든다**

- **명령형 언어는 절차를 지시한다.**
어떻게 반복하고, 검사하고, 값을 갱신할지 하나씩 적는다.
- **선언형 언어는 원하는 결과만 말한다.**
SQL은 어떤 데이터를 원하고 어떻게 변환할지만 적으면 된다.
- **선언형은 DB 최적화기에 유리하다.**
인덱스, 조인 순서, 실행 순서를 DB가 자동으로 고를 수 있다.
- **선언형은 내부 저장 방식 변화에 더 강하다.**
순서나 물리적 배치가 바뀌어도 질의를 안 고치고 성능 개선이 가능하다.
- **선언형은 병렬 실행에도 더 잘 맞는다.**
결과 패턴만 지정하므로 여러 코어/장비에서 실행 전략을 더 자유롭게 짤 수 있다.

## (1) 웹에서의선언형질의

- “선언형이 왜 좋은지”를 DB 말고, 웹 브라우저 예시
- 선언형 방식이 데이터베이스에서만 좋은 게 아니라, 웹에서도 똑같이 장점이 있다
- 이 부분은 CSS와 JavaScript를 비교해서, 
선언형 언어는 “무엇을 원하는지”만 말하게 하고 엔진이 자동으로 적용·최적화하게 해주므로, 
SQL 같은 선언형 질의 언어의 장점을 이해하기 쉽게 보여주는 예시

- 선언형은 **패턴/조건**을 적고,
명령형은 **절차/순서**를 적는다.
- CSS 선택자 `li.selected > p` 는
“selected인 li의 직계 자식 p”라는 패턴을 선언하는 것이다.
- JavaScript DOM 코드는 같은 일을 할 수 있지만,
길고 복잡하고 상태 변화 대응도 수동이며 유지보수가 불편하다.
- 선언형 언어는 엔진(브라우저/DB)이 내부적으로 최적화할 여지를 준다.
사용자 코드를 안 바꾸고도 성능 개선이 가능하다.
- 이 웹 예시는 결국 SQL의 장점을 설명하기 위한 비유다.
CSS가 JS 명령형 스타일링보다 낫듯, SQL도 명령형 질의 API보다 낫다는 뜻이다.

## (2) 맵리듀스 질의

- **맵리듀스(MapReduce) 뜻**
- **왜 결국 MongoDB도 다시 선언형 집계 파이프라인 쪽으로 갔는지**
- **“문서들을 하나씩 읽어서 중간 결과를 뿌리고(map), 같은 것끼리 모아서 합치는(reduce)” 방식**
- 대량 데이터를 여러 컴퓨터에서 처리하기엔 강력하지만,
**SQL보다 더 저수준이고, 쓰기 불편하고, 최적화 여지도 적어서** 
결국 MongoDB도 더 선언적인 **aggregation pipeline**을 추가했다

- 맵리듀스는 **대량 데이터를 여러 컴퓨터에서 처리하기 위한 프로그래밍 모델**이다.
문서를 하나씩 읽어 중간 결과를 뿌리고(map), 같은 키끼리 합친다(reduce).
- 예제에서 map은 각 상어 관찰 문서를 `"연-월" → 마릿수` 형태로 바꾸고,
reduce는 같은 달의 마릿수를 모두 합친다.
- map/reduce 함수는 순수 함수여야 해서 분산 실행, 재실행, 장애 복구에 유리하다.
- 하지만 맵리듀스는 SQL보다 더 저수준이고,
함수 두 개를 조심스럽게 맞춰 작성해야 해서 사용성이 떨어진다.
- 선언형 질의 언어가 최적화에 더 유리하기 때문에
MongoDB도 결국 aggregation pipeline 같은 선언형 집계 언어를 추가했다.
- 즉, NoSQL도 데이터를 다루다 보면 결국 SQL 비슷한 선언형 질의 방식을 다시 만들게 된다.

# [3] 그래프형 데이터모델

- **데이터 사이의 연결이 너무 많아지면, 표나 JSON 트리보다 그래프로 보는 게 더 자연스럽다**
- 즉,  문서 모델 = 트리 구조에 강함,
- 관계형 모델 = 조인으로 관계 처리 가능에서 한 단계 더 가서,
- **관계 자체가 주인공인 데이터는 아예 그래프로 모델링하자**

- 문서 모델은 트리형 데이터에 자연스럽다
한 객체 아래 하위 정보가 달리는 구조에 좋다.
- 관계형 모델은 다대다 관계도 처리할 수 있다
조인으로 꽤 많은 문제를 푼다.
- 하지만 데이터 연결이 매우 복잡하면
그래프로 보는 편이 더 자연스럽다.
- 그래프 모델은
**“연결이 복잡한 데이터의 자연스러운 표현”**
    - 데이터를 “객체들”보다 “객체들 사이의 관계망”으로 보는 방식이며, 
    사람·장소·이벤트처럼 여러 종류의 엔티티가 복잡하게 연결된 데이터를 표현하고 질의할 때

- 그래프 모델은 **정점(객체)** 과 **간선(관계)** 으로 데이터를 표현한다.
- 소셜 그래프, 웹 그래프, 도로/철도 네트워크처럼
**연결 자체가 핵심인 데이터**에 잘 맞는다.
- 그래프는 같은 종류 객체만이 아니라
사람, 장소, 이벤트, 댓글처럼 **서로 다른 타입 객체를 함께 연결**할 수 있다.
- 사람 → 도시 → 국가 → 대륙 같은 식으로
**관계를 여러 번 따라가는 질의**가 자연스럽다.
- 데이터 사이 연결이 단순하면 문서형/관계형으로도 충분할 수 있지만,
**연결망이 복잡해질수록 그래프 모델이 더 직관적**이다.

## (1) 속성그래프

- 그래프형 데이터 모델 중에서도 “속성 그래프(property graph)”가 어떻게 생겼는지
- **그래프 = 점(정점) + 선(간선)** 인데,
- 속성 그래프는 거기에 **점에도 정보 붙이고, 선에도 정보 붙일 수 있는 그래프**

- 속성 그래프는 점과 선으로만 끝나는 게 아니라,
점에도 속성이 있고 선에도 속성이 있다.
- 그래서 현실의 다양한 객체와 다양한 관계를
하나의 연결망 안에 담기 좋다.
- 특히 관계를 따라가며 찾는 문제,
그리고 나중에 새로운 관계를 계속 추가해야 하는 문제에 강하다.

- **정점은 식별자, 들어오는/나가는 간선, 속성을 가진다.**
- **간선은 시작점, 끝점, 관계 이름(label), 속성을 가진다.**
- **속성 그래프는 사람·도시·국가·알레르기처럼 서로 다른 타입을 한 그래프 안에 넣을 수 있다.**
- **관계를 따라 순회하는 질의에 강하다.**
- **새 기능이 생겨도 노드와 간선만 추가하면 되어 확장성이 좋다.**

## (2) 사이퍼 질의언어

- **그래프 DB에서 SQL 비슷하게 질의하는 언어가 Cypher(사이퍼)**
- Cypher는 그래프를 “표”처럼 조회하는 게 아니라,
점과 선이 어떻게 이어져 있는지 그 “모양(pattern)” 자체를 써서 찾는 언어

- Cypher는 그래프에서 노드와 간선의 연결 패턴을 직접 그리듯이 써서 질의하는 선언형 언어이며, 
특히 `WITHIN*0..` 같은 문법으로 몇 단계일지 모르는 관계 경로도 자연스럽게 표현할 수 있다

- Cypher는 그래프용 **선언형 질의 언어**다.
    - “어떻게 찾을지”보다 “어떤 연결 패턴을 찾을지”를 쓴다.
- 괄호 `()` 는 노드, 대괄호 `[]` 는 간선, 화살표 `->` 는 관계 방향을 뜻한다.
- `MATCH` 는 그래프 안에서 특정 패턴의 연결을 찾는 문이다.
- `[:WITHIN*0..]` 은 WITHIN 관계를 0번 이상 반복해서 따라가는 뜻이다.
    - 길이가 가변적인 경로를 표현한다.
- 그래프 질의는 이런 식으로 **관계를 여러 번 따라가야 하는 문제**를 자연스럽게 표현할 수 있다.
- Cypher는 선언형이므로 실행 방법은 DB가 정하고, 사용자는 패턴만 쓰면 된다.

## (3) SQL 의그래프질의

- 그래프 데이터를 SQL로도 질의할 수는 있는데, 꽤 불편하다
- 할 수는 있다. 그런데 Cypher보다 훨씬 장황하고 어렵다

- 이 부분은 그래프 질의를 SQL로도 구현할 수 있지만, 
가변 길이 관계 탐색을 재귀 CTE로 풀어야 해서 Cypher보다 훨씬 길고 덜 자연스럽다

- 이 SQL은 사실 아래 알고리즘을 SQL로 쓴 거예요.
    1. 미국 찾기
    2. 미국 안에 있는 모든 장소 찾기
    3. 유럽 찾기
    4. 유럽 안에 있는 모든 장소 찾기
    5. 미국 안 장소에서 태어난 사람 찾기
    6. 유럽 안 장소에 사는 사람 찾기
    7. 두 조건을 동시에 만족하는 사람만 남기기
    8. 그 사람 이름 출력하기
- 그래프를 SQL로 다룬다는 건 결국 **관계형 테이블 위에서 그래프 탐색을 흉내 내는 것**에 가까움

- 어려운 이유는 그래프 질의는 **가변 길이 경로 탐색**이 많기 때문이다.
즉 몇 번 조인해야 할지 미리 정할 수 없는 경우가 많다.
- Cypher의 `[:WITHIN*0..]` 는
“WITHIN 간선을 0번 이상 따라가라”를 아주 짧게 표현한다.
- SQL에서는 이런 걸 `WITH RECURSIVE` 로 풀 수 있지만,표현이 길고 장황해진다.

### 1) 트리플저장소와스파클

- **그래프 데이터 (속성그래프와 같은 내용) 를 또 다른 방식으로 표현하는 법**
- 앞에서 본 **속성 그래프**가 “노드와 간선에 속성을 붙이는 방식”이었다면,
- 여기서는 같은 생각을 **주어(subject) - 서술어(predicate) - 목적어(object)** 라는 **문장 형태**로 저장.
- 이걸 **트리플 저장소(triple store)**

- 트리플 저장소는 그래프 데이터를 **주어-서술어-목적어 문장들**로 저장한다.
- 목적어는: 숫자/문자열 같은 값일 수도 있고 다른 노드일 수도 있다.
- 그래서 속성 그래프의 노드 속성 노드 간 관계를 모두 표현할 수 있다.
- Turtle은 이런 트리플을 사람이 읽기 좋게 적는 문법/ 표기법

### 2) 시맨틱웹

- **트리플 저장소**
- **시맨틱 웹**
- 둘은 관련은 있지만, 같은 건 아님

- 시맨틱 웹은 웹의 정보를 컴퓨터도 이해할 수 있는 데이터로 공개하고 연결하자는 아이디어
    - (웹사이트가 사람이 읽는 정보뿐 아니라
    컴퓨터도 읽을 수 있는 구조화된 데이터를 같이 게시하자)
- 트리플 저장소는 그와 별개로도 쓸 수 있는 데이터 모델이며,
- 시맨틱 웹 자체는 과대광고와 복잡성 때문에 크게 성공하지 못했지만 
그 과정에서 나온 트리플/RDF 기술과 아이디어는 여전히 가치가 있다

### 3) RDF 데이터 모델

- 트리플 저장소/RDF를 “사람이 읽기 쉬운 표기”와 “기계끼리 교환하기 좋은 표기”로 어떻게 표현하느냐
- RDF의 본질은 여전히 “주어-서술어-목적어” 트리플인데,
- Turtle은 사람이 읽기 쉽게 쓴 버전이고,
- RDF/XML은 XML 문법으로 감싼 버전
    - RDF는 인터넷 전체의 데이터 교환을 염두에 두었기 때문에
    단어 대신 **URI/네임스페이스**를 써서 의미 충돌을 피하려 한다.
- 여기서 URI는 꼭 접속 가능한 URL이 아니라, 전역적으로 안 겹치는 **고유한 이름표** 역할도 함

### 4) 스파클질의언어

- **SPARQL이 뭔지**
- **그래프 DB가 옛날 코다실(CODASYL) 네트워크 모델이랑 왜 비슷해 보이지만, 실제로는 왜 다른지**

- **SPARQL은 RDF 트리플 저장소용 선언형 질의 언어**다.
- `?person :bornIn / :within* ?location` 같은 문법으로 **관계를 여러 단계 따라가는 패턴**을 간결하게 쓸 수 있다.
- RDF는 속성과 간선을 구분하지 않고 둘 다 predicate로 보기 때문에, SPARQL 문법이 더 통일적이다.
- 그래프 DB는 코다실 네트워크 모델과 달리 **더 유연한 스키마, 직접 노드 참조, 정렬 비의존성, 선언형 질의 언어**를 가진다.
- 그래서 그래프 DB는 **코다실이 다시 온 것**이 아니라, 관계망 데이터를 훨씬 현대적으로 다루는 모델이다.

### 5) 초석: 데이터로그

- **데이터로그(Datalog)** 가 뭔지
- **“데이터를 바로 긁어오는 언어”라기보다,
작은 사실(fact)들과 규칙(rule)을 쌓아서
새로운 사실을 추론해내는 언어**
- Cypher, SPARQL은: “이 패턴을 만족하는 것 찾아줘”에 가까웠다면,
- 데이터로그는: 이런 사실들이 있고, 이런 규칙을 적용하면, 이런 새 사실도 성립한다
를 단계적으로 만들어가는 방식

- **데이터로그는 fact(사실)와 rule(규칙)로 동작한다.**
- **트리플 저장소와 비슷하지만 `(주어, 서술어, 목적어)` 대신 `서술어(주어, 목적어)` 형태로 쓴다.**
- **`within_recursive` 는 재귀 규칙으로, 어떤 장소가 더 큰 지역 안에 포함되는지 반복적으로 추론한다.**
- **`migrated` 는 출생지와 거주지를 조합해 “어디서 태어나 어디 사는 사람”이라는 새 사실을 만든다.**
- **간단한 질의엔 덜 편하지만, 규칙을 결합·재사용할 수 있어서 복잡한 데이터에는 매우 강력하다.**
