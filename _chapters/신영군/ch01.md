# 4. Specific Application Cases & Concept Review

## 4.1 Twitter example and Load Parameter

When I comes to this section, the example is kind of confusing, but i figured out that twitter faced the problem of overload

```markdown
fan-out:each user follows many people, and each user is followed by many people.
```

In method 2, twitter chooses to save the new-posted tweet to the home-timeline of all followers. This is called **fan-out on write**. But for celebrities, it is not efficient to use this method. So twitter chooses to use **fan-out on read** for celebrities. Then the following concept is emerged:

```markdown
load-parameter: It is a metric, describing the load of the system.
```

in this example, load parameter is being set as **distribution of followers per user**, and this metric can be changed depending on the application. e.g. requests per second to a web server, the ratio of reads to writes in a database.
Twitter measures users' type by the load parameters, and if the user is a celebrity, here is a step-by-step breakdown of what Twitter did:

1. The Request: When you open Twitter, your app asks the system for your home timeline.
2. Fetching the Cache: The system immediately grabs your pre-computed cache, which is already filled with the fanned-out tweets from the "normal" people you follow.
3. Fetching the Celebrities: At that exact same time, the system looks up any celebrities you follow, finds their recent tweets from a global collection, and pulls them down.
4. The "Merge": Before sending the data to your screen, the system takes the tweets from your cache and the newly fetched celebrity tweets and combines them together, sorting them by time so they appear in the correct chronological order

## 4.2 Reasons for using percentile to measure performance instead of average

Since the average is easily affected by the outliers, it is not a good metric to measure the performance of the system. Instead, we should use the percentile to measure the performance of the system. For example, if we observe the 50th percentile, it means that 50% of the requests are processed in this time. If we observe the 95th percentile, it means that 95% of the requests are processed in this time. And we can utilize this robust metric to conduct further business analysis. Research reveals that average 100ms latency increment leads to 1% revenue drop.

## 4.3 Agile development strategy

The author discusses the Agile strategy—a methodology prioritizing flexibility and rapid iteration over the linear, sequential Waterfall model. However, as noted in academic examples, developers often face significant pressure due to this pace, frequently resulting in technical debt or sprint overflows. To mitigate this, teams utilize Daily Stand-ups to synchronize progress and rely on Project Management Dashboards (such as Jira or Kanban boards) to monitor defects and maintain velocity.
