## 데이터 시스템

- 요즘 앱은 CPU보다 **데이터가 병목**임 (양, 복잡도, 변화 속도)
- 데이터 시스템 구성요소: DB, Cache, Search Index, Stream Processing, Batch Processing
- 근데 요즘은 이 경계가 많이 흐려짐
	- Redis를 메시지 큐처럼 쓰기도 하고, Kafka가 DB급 내구성을 제공하기도 함
	- 결국 여러 도구 조합해서 하나의 데이터 시스템을 만드는 게 일반적 → 개발자가 데이터 시스템 설계자 역할까지 하게 됨

이 장에서 다루는 세 가지 축: **Reliability, Scalability, Maintainability**

---

## 1. Reliability (신뢰성)

뭔가 잘못돼도 시스템이 계속 동작해야 한다는 것

- **fault ≠ failure**
	- fault: 시스템 일부가 스펙에서 벗어남
	- failure: 시스템 전체가 죽는 것
	- 핵심은 fault가 failure로 번지지 않게 하는 것
	- Netflix Chaos Monkey — 일부러 결함 주입해서 내결함성 테스트하는 방식도 있음

### 하드웨어 결함
- 하드디스크 MTTF가 10~50년이라고 해도, 디스크 1만개짜리 클러스터면 매일 1개씩 고장남
- 전통적 대응: RAID, 이중 전원, 핫스왑 CPU 등 하드웨어 중복
- 근데 클라우드(AWS 등)에서는 VM이 그냥 죽을 수 있어서 → SW 레벨 내결함성이 더 중요해짐
- 이렇게 하면 rolling upgrade(노드 하나씩 패치)도 가능

### 소프트웨어 오류
- HW 결함은 보통 독립적으로 발생하는데, SW 버그는 노드 간 상관관계가 있어서 더 위험
- 예시들:
	- 2012년 윤초 → 서버 동시 크래시
	- 폭주 프로세스가 공유 자원 다 잡아먹음
	- 의존 서비스 느려지거나 이상한 응답
	- cascading failure (연쇄 장애)
- 딱히 은총알은 없고 테스트 잘 하고, 프로세스 격리하고, 모니터링 열심히 하는 수밖에

### 인간 오류
- 사실 대형 장애 원인 대부분은 사람임 (하드웨어는 10~25% 정도)
- 대응:
	- 실수하기 어렵게 API/인터페이스 설계
	- sandbox 환경
	- 테스트 자동화
	- 빠른 롤백, canary release
	- 모니터링/텔레메트리

### 신뢰성이 중요한 이유
- "우리 서비스는 핵심 아니니까 괜찮다"는 안 됨 — 사용자 데이터 날리면 책임은 우리한테 옴
- 비용 때문에 신뢰성 타협할 수는 있지만, 그 트레이드오프를 알고 있어야 함

## 2. Scalability (확장성)

지금 잘 돌아간다고 나중에도 잘 돌아가는 건 아님. 유저 1만 → 10만 → 100만 되면 시스템도 바뀌어야 함.

"X는 확장 가능하다" 이런 식으로 말하는 건 의미 없고, "부하가 이만큼 늘면 우리 선택지가 뭔가?"가 맞는 질문

### 부하 기술 (Describing Load)

부하를 숫자로 표현해야 함 → **load parameter** (초당 요청 수, 읽기/쓰기 비율, 캐시 적중률 등)

**Twitter 예시가 좋음:**
- 트윗 작성: 평균 4.6k/초, 피크 12k/초
- 홈 타임라인 조회: 300k/초
- 쓰기 자체는 별거 아닌데 문제는 **fan-out** — 트윗 하나가 팔로워 전원한테 가야 함

- 접근법 1: 그냥 저장하고 타임라인 조회할 때 JOIN → 쓰기 쉽고 읽기 비쌈
- 접근법 2: 트윗할 때 팔로워 각각의 타임라인 캐시에 미리 넣어둠 → 읽기 쉽고 쓰기 비쌈 (팔로워 3천만이면 쓰기 3천만번)
- Twitter 결론: **하이브리드** — 보통 유저는 접근법 2, 팔로워 많은 유명인은 접근법 1

### 성능 기술 (Describing Performance)

부하 정의하고 나면 질문은 두 가지:
1. 리소스 그대로인데 부하 늘리면 성능이 어떻게 되나?
2. 성능 유지하려면 리소스를 얼마나 더 넣어야 하나?

- 배치 시스템 → **throughput** 중요
- 온라인 시스템 → **response time** 중요
- latency ≠ response time: latency는 대기 시간, response time은 네트워크 지연까지 포함한 전체 시간

**백분위수 (percentile)**
- 평균은 별로 쓸모없음. 백분위수를 봐야 함
- p50(중앙값), p95, p99, p999
- tail latency가 중요한 이유: 응답 느린 유저가 오히려 데이터 많은 VIP 고객일 수 있음
	- Amazon 관측: 응답 100ms 느려지면 매출 1% 감소
- SLA 보통 "p50 < 200ms, p99 < 1s" 이런 식으로 정의
- **head-of-line blocking**: 느린 요청 몇 개가 뒤의 빠른 요청들까지 다 막아버림
- **tail latency amplification**: 요청 하나가 백엔드 여러개 호출하는데 그중 하나만 느려도 전체가 느려짐

### 부하 대응

- **Scale up** (수직): 더 좋은 머신. 단순하지만 한계 있음
- **Scale out** (수평): 여러 머신에 분산 (shared-nothing)
- **Elastic**: 부하 따라 리소스 자동 조절
- 실제로는 섞어 씀. stateless 분산은 쉬운 편인데 stateful은 복잡
- 범용 정답 같은 건 없음 — 읽기/쓰기 패턴, 데이터 크기, 응답 시간 요구사항에 따라 아키텍처가 다 다름

## 3. Maintainability (유지보수성)

소프트웨어 비용 대부분은 초기 개발이 아니라 유지보수에서 나옴 (버그 수정, 장애 대응, 기능 추가, 기술 부채 등)

### 운용성 (Operability)
- "좋은 운영이 나쁜 SW를 커버할 수 있지만, 좋은 SW도 나쁜 운영에선 못 버팀"
- 운영팀: 모니터링, 장애 복구, 원인 추적, 보안 패치, 용량 계획 등
- 시스템이 도와줄 수 있는 것:
	- 좋은 모니터링 (내부 상태 가시성)
	- 자동화, 표준 도구 연동
	- 특정 머신에 의존하지 않는 구조
	- 예측 가능하게 동작

### 단순성 (Simplicity)
- 프로젝트 커지면 복잡도 폭발함 — 상태 공간 커지고, 모듈간 커플링 강해지고, 의존성 꼬이고, 네이밍 엉망되고, 성능 핵 쌓이고
- 복잡하면 유지보수 비용 올라가고 버그 생기기 쉬움
- 해결책: **추상화** — 복잡한 구현을 깔끔한 인터페이스 뒤에 숨기기 (프로그래밍 언어가 머신코드 숨기고, SQL이 디스크 구조 숨기는 것처럼)
- 핵심은 **우발적 복잡성**(accidental complexity)을 없애는 것. 문제 자체의 복잡성은 어쩔 수 없음

### 진화성 (Evolvability)
- 요구사항은 계속 바뀜 (비즈니스 우선순위, 유저 요구, 규제 등)
- Agile(TDD, 리팩토링)이 변화 대응 프레임워크를 주긴 하는데 그건 코드 레벨 얘기
- 이 책에서는 데이터 시스템 수준의 민첩성을 **evolvability**라고 부름
- 결국 단순한 시스템이 고치기도 쉬움 → 단순성과 진화성은 연결돼 있음
