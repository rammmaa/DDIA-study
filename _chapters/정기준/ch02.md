## 1. Relational vs. Document
### The Impedance Mismatch
Most application development today is done in object-oriented languages. However, if data is stored in relational tables, an awkward translation layer is required (ORM).
* **Relational Approach:** object into multiple tables. Requires complex joins to reconstruct.
* **Document Approach:** Stores data in a tree structure (JSON). Provides better **schema flexibility** and **storage locality**.

### Schema-on-Read vs. Schema-on-Write
* **Schema-on-Write (Relational):** The database ensures all data conforms to a specific structure before it is stored. 
* **Schema-on-Read (Document):** The database is indifferent to the structure; the application interprets the data format when it is retrieved.

---

## 2. Relational: Many-to-One and Many-to-Many
Relational databases shine when data has complex relationships.

* **Normalization:** By storing "Region ID" instead of the string "Greater London" in every user row, we avoid data redundancy. If the region name changes, we update it in exactly one place.
* **The Join Problem:** Document databases struggle here. If you need to reference a piece of data that is shared across many documents, you either have to:
    1.  **Denormalize:** Duplicate the data (risks inconsistency).
    2.  **Application-side Joins:** Make multiple queries to the DB and merge results in code (slow).

## 3. Query Languages: Declarative vs. Imperative
The shift from IMS and CODASYL to SQL was a shift in philosophy.

* **Imperative (How):** You tell the computer to perform specific operations in a specific order (e.g., `for` loops). If the data grows, your specific "path" might become inefficient.
* **Declarative (What):** You describe the pattern of data you want (SQL). This is powerful because the **Query Optimizer** can automatically improve performance (by picking indexes or changing join orders) without you changing a single line of code.

---

## 4. Graph-Like Data Models
When relationships between entities are as important as the entities themselves, a graph model is often more natural than a relational one.

### The Property Graph 
In this model, every node and relationship can have its own properties 
* **Edges are directed:** They have a head and a tail, but the graph can be traversed in either direction.
* **Flexibility:** You can easily add new types of relationships between existing nodes without migrating a schema.
