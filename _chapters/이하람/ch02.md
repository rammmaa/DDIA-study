### 데이터 모델과 질의 언어
* **데이터 모델의 중요성**
	* 데이터 모델은 소프트웨어가 어떻게 작성되는지뿐 아니라, 문제를 어떻게 바라보고 해결할지에도 영향을 준다.
	* 대부분의 애플리케이션은 여러 데이터 모델을 계층적으로 쌓아 만든다.
		* 현실 세계 → 객체/데이터 구조/API
		* 저장 계층 → JSON/XML, 관계형 테이블, 그래프 모델
		* 저장소 내부 → 메모리/디스크/네트워크 상의 바이트 표현
	* 각 계층은 하위 계층의 복잡성을 숨기기 위한 추상화 역할을 한다.
	* 데이터 모델마다 잘하는 일과 못하는 일이 다르므로 문제에 맞는 모델을 고르는 것이 중요하다.

### 01. 관계형 모델과 문서 모델
* **관계형 모델**
	* 관계형 모델은 데이터를 테이블(관계)과 로우(튜플)로 표현한다.
	* SQL은 관계형 모델의 대표적인 질의 언어다.
	* 관계형 데이터베이스는 구현 세부를 숨기고, 정규화된 구조로 데이터를 저장/질의하는 데 강점이 있다.
	* 조인, 다대일 관계, 다대다 관계를 다루는 데 유리하다.

* **NoSQL의 등장**
	* NoSQL은 관계형 모델만으로 해결하기 어려운 요구를 다루기 위해 등장했다.
	* 주요 배경
		* 대규모 데이터셋, 높은 쓰기 처리량
		* 오픈소스 선호
		* 특수한 질의 패턴
		* 더 유연한 데이터 모델에 대한 요구
	* 하지만 관계형 DB를 완전히 대체한다기보다, 함께 사용되는 경우가 많다.
	* 이를 **다중 저장소 지속성(polygot persistence)** 이라고 한다.

* **객체-관계형 불일치**
	* 객체지향 언어의 객체 구조와 관계형 테이블 구조는 잘 맞지 않는 경우가 많다.
	* 이를 **임피던스 불일치(impedance mismatch)** 라고 한다.
	* ORM은 이를 완화하지만 완전히 없애지는 못한다.

* **문서 모델**
	* 문서 모델은 관련된 데이터를 하나의 문서(JSON 등) 안에 묶어 저장한다.
	* 이력서, 프로필처럼 하나의 문서 단위로 읽고 쓰기 좋은 데이터에 적합하다.
	* 장점
		* 애플리케이션 객체 구조와 더 비슷함
		* 관련 정보가 한곳에 있어 지역성(locality)이 좋음
		* 일대다 트리 구조를 자연스럽게 표현 가능
	* 단점
		* 조인 지원이 약한 경우가 많음
		* 다대일/다대다 관계가 많아질수록 불편해짐
		* 문서 일부만 필요해도 전체 문서를 읽어야 할 수 있음

* **정규화와 ID 참조**
	* 사람이 읽는 문자열 대신 ID를 저장하면 중복을 줄일 수 있다.
	* 의미 있는 정보는 한 곳에만 저장하고, 다른 곳에서는 ID로 참조한다.
	* 이는 **정규화(normalization)** 의 핵심 아이디어다.
	* 관계형 모델은 이런 참조와 조인에 강하다.
	* 문서 모델은 트리 구조에는 강하지만, 상호 연결된 데이터에는 약할 수 있다.

* **어떤 모델이 더 적합한가**
	* 데이터가 **문서처럼 생긴 일대다 트리 구조** 라면 문서 모델이 적합하다.
	* 데이터 간 **다대다 관계와 상호 연결** 이 많다면 관계형 모델이 더 적합하다.
	* 즉, 어느 모델이 무조건 더 좋다기보다 **데이터의 관계 구조** 가 선택 기준이 된다.

* **문서 모델에서의 스키마 유연성**
	* 문서 데이터베이스는 보통 스키마를 강제하지 않는다.
	* 이를 흔히 **스키마리스(schemaless)** 라고 하지만, 실제로는 읽는 쪽에서 암묵적으로 구조를 가정하는 경우가 많다.
	* 더 정확히는
		* **쓰기 스키마(schema-on-write)**: 관계형 DB 중심
		* **읽기 스키마(schema-on-read)**: 문서 모델 중심
	* 구조가 자주 바뀌거나 데이터 형태가 일정하지 않다면 문서 모델이 유리할 수 있다.
	* 반대로 구조가 일정하고 예측 가능하다면 명시적 스키마가 더 유용하다.

* **문서 DB와 관계형 DB의 경계**
	* 최근에는 관계형 DB도 JSON/XML 지원을 강화하고 있고,
	* 문서 DB도 참조, 조인 비슷한 기능을 일부 지원한다.
	* 즉 둘은 점점 서로의 장점을 흡수하며 비슷해지고 있다.

### 02. 데이터를 위한 질의 언어
* **선언형 질의 언어 vs 명령형 질의 언어**
	* **명령형**: 데이터를 어떻게 찾을지 절차를 하나씩 직접 지정
	* **선언형**: 어떤 결과를 원하는지만 기술
	* SQL은 선언형 질의 언어의 대표 사례다.
	* 선언형 언어의 장점
		* 더 간결함
		* 구현 세부를 숨길 수 있음
		* DB가 내부적으로 최적화하기 쉬움
		* 병렬 처리에 더 유리함

* **질의 최적화기(query optimizer)**
	* 관계형 데이터베이스는 질의 최적화기가
		* 어떤 순서로 실행할지
		* 어떤 색인을 사용할지
	  를 자동으로 결정한다.
	* 따라서 개발자는 접근 경로를 직접 세세하게 짤 필요가 없다.
	* 이것이 관계형 모델의 큰 장점 중 하나다.

* **맵리듀스(MapReduce)**
	* 맵리듀스는 대량 데이터를 여러 장비에서 처리하기 위한 프로그래밍 모델이다.
	* 일부 NoSQL 시스템은 이를 이용한 질의를 지원했다.
	* 다만 SQL 같은 선언형 질의보다 저수준이라 사용성이 떨어진다.
	* 그래서 많은 시스템이 결국 선언형 집계 언어를 다시 도입하게 되었다.

### 03. 그래프형 데이터 모델
* **그래프 모델이 필요한 경우**
	* 데이터에서 다대다 관계가 흔하고, 연결 구조가 복잡할수록 그래프 모델이 자연스럽다.
	* 예
		* 소셜 네트워크
		* 웹 링크 구조
		* 도로/철도 네트워크
		* 사람-장소-이벤트-조직 간 관계

* **그래프의 기본 요소**
	* **정점(vertex)**: 사람, 장소, 조직 같은 개체
	* **간선(edge)**: 개체들 사이의 관계
	* 그래프 모델은 서로 다른 종류의 개체와 관계를 한 저장소 안에 함께 표현하기 좋다.

* **속성 그래프(property graph)**
	* 정점
		* 고유 식별자
		* 속성(키-값)
		* 유입/유출 간선
	* 간선
		* 고유 식별자
		* 시작 정점, 끝 정점
		* 관계 레이블
		* 속성
	* 장점
		* 관계 구조를 유연하게 확장 가능
		* 상호 연결이 많은 데이터를 표현하기 쉬움
		* 새로운 기능이 생겨도 구조를 비교적 자연스럽게 확장 가능

* **사이퍼(Cypher)**
	* 그래프 데이터베이스(예: Neo4j)에서 쓰는 선언형 질의 언어다.
	* 노드와 간선을 패턴처럼 써서 질의한다.
	* 복잡한 그래프 순회를 비교적 직관적으로 표현할 수 있다.

* **SQL과 그래프 질의**
	* 그래프 데이터도 관계형 테이블로 표현할 수는 있다.
	* 하지만 가변 길이 경로를 따라가는 질의는 SQL에서 복잡해진다.
	* 이 때문에 그래프 질의는 전용 모델/언어가 더 자연스러운 경우가 많다.

* **트리플 저장소(triple store)**
	* 정보를 **(주어, 서술어, 목적어)** 형태로 저장한다.
	* 속성 그래프와 매우 비슷한 생각을 다른 용어로 표현한 모델이다.
	* RDF, 시맨틱 웹과 연결되지만, 꼭 웹 데이터 공개 목적이 아니어도 내부 데이터 모델로 쓸 수 있다.

* **스파클(SPARQL)**
	* RDF/트리플 저장소용 선언형 질의 언어다.
	* 패턴 매칭 방식으로 질의하며, 그래프 질의를 간결하게 표현할 수 있다.

* **데이터로그(Datalog)**
	* 오래된 규칙 기반 질의 언어다.
	* 현대 그래프/질의 언어의 이론적 기반으로 중요하다.
	* 사실(fact)과 규칙(rule)을 기반으로 새로운 사실을 유도하는 방식이다.

* **관계형 모델**
	* 조인, 정규화, 다대다 관계에 강함
	* 구조가 비교적 명확하고 반복적인 데이터에 적합
* **문서 모델**
	* 문서 단위 데이터, 일대다 트리 구조에 강함
	* 유연한 스키마와 지역성이 장점
* **그래프 모델**
	* 상호 연결이 많은 데이터에 가장 자연스러움
	* 복잡한 관계 탐색과 경로 질의에 강함
