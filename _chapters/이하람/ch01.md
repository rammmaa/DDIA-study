### 데이터 시스템에 대한 생각
* objective: 특정 목적에 어떤 기술이 적합한지 결정하고, 좋은 애플리케이션 아키텍처의 기반을 만들기 위해 도구를 조합하는 방법을 이해하는 능력을 기른다
* 데이터 중심적 <> 계산 중심적
	* 문제: 데이터 양, 데이터 복잡성, 데이터가 변히는 속도 등 <> CPU 사이클
* ![스크린샷](/DDIA-study/assets/images/이하람/스크린샷 2026-03-24 오후 3.04.57.png)
* **기능적 요구사항:** 애플리케이션이 처리하는 다양한 작업
* **비기능적 요구사항:** 보안, 신뢰성, 법규 준수, 확장성, 호환성, 유지보수성
### 01. Reliablity
1) 사용자가 기대한 기능 수행(expected)
2) 사용자가 범한 실수나 예상치 못한 소프트웨어 사용법 허용 가능(fault tolerance; resiliency)
3) 예상된 부하와 데이터 양에서 필수적 사용 사례를 충분히 만족
4) 허가되지 않은 접근과 오남용을 방지(protection)
* **하드웨어 장애**
	* 시스템 장애율을 줄이기 위한 대응으로는 각 하드웨어 구성 요소에 중복을 추가하는 것이 있다. 데이터 양과 애플리케이션 부하가 크지 않은 경우, 단일 장비의 전체 장애는 드물게 일어나기 때문에 하드웨어 구성 요소의 복제로 시스템 장애율을 크게 줄일 수 있었다. 하지만 데이터 양과 애플리케이션 계산 요구가 늘어나면서 애플리케이션 규모와 장비 수가 증가, 결과적으로 하드웨어 결함률도 증가했다.
* **소프트웨어 장애**
	* 따라서 상술한 문제를 해결하기 위해 소프트웨어 내결함성 기술을 사용하거나 하드웨어 중복성을 추가해 전체 장비의 손실을 견딜 수 있는 시스템으로 옮겨가고 있다. 장비 장애를 견딜 수 있는 시스템은 전체 시스템의 중단시간 없이 한 번에 한 노드씩 패치가 가능하다.
	* 정교한 테스트, process isolation, 재시작 허용 정책, 시스템 동작 측정, 모니터링과 같은 것들과 일부 지표를 보장하는 것이 장애를 예방하는 것에 도움을 줄 수 있다. 
* **인적 오류**
	* 인적 오류를 최소화하는 방법은 오류의 가능성을 최소화하는 방향으로 시스템을 설계하는 것이다. 잘 설계된 추상화, API, 관리 인터페이스를 사용하는 것이 유용하다. 장애 발생의 영향을 최소화하기 위해 인적 오류를 빠르고 쉽게 복구할 수 있도록 해라. 설정 변경 내역을 빠르게 롤백하고 새로운 코드를 롤아웃하거나, 이전 계산이 잘못된 경우를 대비해 데이터 재계산 도구를 제공하라. 또한 성능 지표와 오류율 같은 상세하고 명확한 모니터링 대첵을 마련하는 것이 중요하다.
### 02. Scalablity
* 확장성은 증가한 부하에 대처하는 시스템 능력을 설명하는 데 사용하는 용어다. 부하는 부하 매개변수로 나타낼 수 있으며, 대표적인 지표로 웹 서버의 초당 요청 수, 데이터베이스 읽기 대 쓰기 비율, 대화방의 동시 활성 사용자, 캐시 적중률 등이 될 수 있다. 
	* e.g. twitter
		* 트윗 작성은 간단히 새로운 트윗을 트윗 전역 컬렉션에 삽입한다. 사용자가 자신의 홈 타임라인을 요청하면 팔로우하는 모든 사람을 찾고, 이 사람들의 모든 트윗을 찾아 시간순으로 정렬해서 합친다.
		* 각 수신 사용자용 트윗 우편함처럼 개별 사용자의 홈 타임라인 캐시를 유지한다. 사용자가 트윗을 작성하면 해당 사용자를 팔로우하는 사람을 모두 찾고 팔로워 각자의 홈 타임라인 캐시에 새로운 트윗을 삽입한다. 그러면 홈 타임라 인의 읽기 요청은 요청 결과를 미리 계산했기 때문에 비용이 저렴하다.
		* 사용자당 팔로워의 분포는 팬아웃 부하를 결정한다. 이 경우 확장성을 논의할 때 핵심 부하 매개변수가 되는 것은 사용자당 팔로워 분포이다. 결론적으로 트위터는 두 접근 방식의 혼합형으로 운영하여, 팔로워가 매우 많은 사용자의 경우 팬 아웃에서 제외되고 1과 같은 방식으로 사용자의 홈 타임라인에 합친다.
	* e.g. Haddop
		* 처리량(throughput) / 응답 시간(respomse time)
		* vs. latency: 응답 시간은 요청을 처리하는 실제 시간 외에도 네트워크 지연과 큐 지연을 포함한다. 지연 시간은 요청이 처리되기를 기다리는 시간으로, 서비스를 기다리며 휴지 상태인 시간을 말한다. 따라서 응답 시간이 latency보다 항상 더 길다.
		* 응답 시간은 단일 숫자가 아닌 측정 가능한 값의 분포로 생각해야 한다. 그라운드 프로세스의 컨텍스트 스위치 (context switch), 네트워크 패킷 손실과 TCP 재전송, 가비지 컬렉션 휴지(garbage collection pause), 디스크에서 읽기를 강제하는 페이지 폴트(p ag e fault), 서버 랙의 기계적인 진동 등으로 추가 지연이 생길 수 있기 때문이다.
		* 산술 평균 / 백분위 / **중앙값**
		* 백분위: 서비스 수준 목표(service level objective, SLO), 서비스 수준 협약서(service level agreement, SLA)에 주로 사용한다.
		* 선두 차단: 소수의 느린 요청 처리만으로 후속 요청 처리가 지체된다.
* **용량 확장(scaling up) / 수직 확장(vertical scaling) vs 규모 확장(scaling out) / 수평 확장(hori zontal scaling)**
* 부하 증가를 감지하면 컴퓨팅 자원을 자동으로 추가할 수 있는 특성을 탄력성(elasticity)이라고 부른다. 다수의 장비에 stateless 서비스를 배포하는 일은 간단하지만, 단일 노드에 stateful한 데이터 시스템을 분산 설치하는 일은 복잡도가 추가적으로 발생한다. 
### 03. Maintainablity
* **운용성(operability)**
	* 운영에 있어서 편리해야 한다. 즉, 운영 중 일부 측면을 자동화할 수 있고 또 자동화해야 한다.
* **단순성(simplicity)**
	* 상태 공간의 급증, 모듈 강 강한 커플링, 복잡한 의존성, 일관성 없는 명명과 용어, 성능 문제 해결을 목표로 한 해킹, 임시방편으로 문제 해결 등으로 발생하는 복잡성을 없애야 한다. 복잡한 소프트웨어에서 변경이 있을 때 버그가 생길 위험이 더 크기 때문이다. 
	* 우발적 복잡도를 제거하기 위한 최상의 도구는 추상화다. 애플리케이션의 특정 요소를 추상화함으로써, 직관적 외관 뒤로 세부 구현을 숨기면서 다른 애플리케이션에서도 사용 가능하도록 할 수 있다. 
* **발전성(evolvability); extensibility, modifiability, plasticity**
	* 시스템의 요구사항은 끊임없이 변한다. 그러므로 조직 프로세스의 측면에서 agile한 작업 패턴은 변화에 적응하기 위한 프레임워크를 제공한다.
